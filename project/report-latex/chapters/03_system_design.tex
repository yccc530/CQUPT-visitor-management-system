\section{系统设计}

\subsection{概念结构设计}

\paragraph{1. 设计目标}

概念结构设计的任务是将需求分析阶段得到的业务对象、数据流、数据存储和处理过程抽象为数据库概念模型。本阶段不直接编写 SQL 建表语句，而是从“重庆邮电大学智慧访客预约与出入校管理系统”的业务语义出发，识别实体、属性、主键候选和实体之间的联系，为后续逻辑结构设计和 MySQL 表结构设计提供依据。

系统核心业务流程为：访客提交预约申请 → 系统检查黑名单 → 被访人确认 → 部门审批 → 生成通行凭证 → 门岗核验 → 入校登记 → 离校登记 → 访问记录归档 → 查询统计。概念模型必须能够覆盖该流程中的参与者、业务单据、审批轨迹、通行凭证、出入校记录、安全控制和统计支撑数据。

\paragraph{2. 实体识别方法}

根据需求分析和数据字典，系统中的实体主要来源于以下几类对象：

参与业务的人或组织：系统用户、角色、权限、部门、访客。\\
校园出入管理的基础对象：校门、访客车辆、随行人员。\\
核心业务单据：访客预约、审批记录、通行凭证、出入校记录、黑名单。\\
系统运行支撑对象：通知消息、操作日志、字典类型、字典项。\\
自动化交付支撑对象：截图记录、报告记录。\\

在概念结构中，用户与角色、角色与权限属于多对多联系，后续逻辑结构设计将转换为 `sys\_user\_role` 和 `sys\_role\_permission` 两个关系表。

\paragraph{6. 概念结构设计结论}

本阶段识别的实体能够覆盖系统从预约提交、审批核验、出入校登记到统计归档的完整业务流程。用户、角色和权限实体能够支撑 RBAC 权限控制；预约、审批、通行凭证和出入校记录实体能够支撑核心业务状态流转；部门、校门、黑名单和日志实体能够支撑校园管理、安全控制和统计分析；截图记录和报告记录实体则支撑本课程设计要求中的自动截图与自动报告生成。

该概念模型不直接等同于 SQL 表结构，但已经为后续逻辑结构设计提供了清晰的实体、属性和联系基础。多对多联系将在逻辑设计阶段转换为关联表，实体属性将在 MySQL 表字段中进一步细化数据类型、约束、索引和默认值。
\paragraph{7. E-R 图表达策略}

由于本系统同时包含访客预约主流程、RBAC 权限控制、消息日志、字典配置、截图与报告生成等多类实体，如果将全部实体和联系绘制在一张 E-R 图中，会导致图形过大、连线交叉严重，不利于课程设计报告阅读。因此，本项目在概念结构设计阶段采用“总体简化图 + 分域详图”的表达方式：

`er\_overview.mmd` 用于展示系统主要实体和主干关系，适合放在报告正文开头说明整体概念模型。\\
`er\_core\_business.mmd` 用于展示访客预约、审批、通行码、门岗核验、出入校登记和黑名单检查等核心业务实体，适合作为正文重点图。\\
`er\_user\_permission.mmd` 用于展示部门、用户、角色、权限及关联表，说明系统的 RBAC 权限模型。\\
`er\_system\_support.mmd` 用于展示通知、操作日志、数据字典、截图记录和报告记录等支撑实体。\\
`table\_relation.mmd` 用于展示后续逻辑结构中的主外键关系，适合作为其它设计图或附录。\\

这种拆分方式既保留了完整的概念设计信息，又避免在同一张图中强行展示所有实体造成的视觉混乱。后续逻辑结构设计仍然以完整实体清单为基础，多对多联系转换为关联表，弱关联和支撑性实体在表关系图中补充说明。

\subsection{逻辑结构设计}

\paragraph{1. 设计目标}

逻辑结构设计是在概念结构设计基础上，将实体、属性和联系转换为适合 MySQL 8 实现的关系模式。本阶段重点明确表、字段、主键、外键、唯一约束、非空约束、状态字段、索引建议和范式检查，为后续 `schema.sql` 建表脚本提供直接依据。

\paragraph{2. 转换原则}

每个独立实体转换为一张关系表，表名统一使用小写下划线命名。\\
主键统一使用 `id`，类型建议为 `BIGINT`。\\
一对多联系通过在“多”的一方增加外键实现。\\
多对多联系转换为关联表，例如 `sys\_user\_role`、`sys\_role\_permission`。\\
时间字段统一使用 `create\_time`、`update\_time`，业务时间保留明确名称，如 `entry\_time`、`approval\_time`。\\
如需软删除，统一使用 `deleted` 字段，`0` 表示未删除，`1` 表示已删除。\\
状态字段统一使用 `VARCHAR(30)`，具体取值后续由字典表或业务常量维护。\\
关键查询字段设置索引，保证预约查询、审批查询、门岗核验和统计报表效率。\\

\paragraph{5. 索引设计思路}

登录认证：`sys\_user.username`、`sys\_user.phone`。\\

审批查询：`approval\_record.apply\_id`、`approver\_user\_id`、`approval\_result`。\\

\paragraph{6. 范式检查}

\subparagraph{6.1 第一范式}

各表字段均为原子字段，例如访客手机号、证件号、预约开始时间、审批结果、通行码等都不再拆分为重复组。随行人员和车辆信息没有直接以逗号拼接方式存储在预约表中，而是分别使用 `visitor\_companion` 和 `visitor\_vehicle` 表管理。

\subparagraph{6.2 第二范式}

所有业务表使用单字段主键 `id`，非主属性均依赖该主键。多对多关系通过关联表独立表示，避免用户角色和角色权限在主表中形成部分依赖。

\subparagraph{6.3 第三范式}

部门名称、校门名称、角色名称、权限名称等独立信息不在预约表、审批表和出入校记录表中重复保存，而是通过外键关联获取。这样可以减少冗余并保证基础数据修改后的一致性。

\subparagraph{6.4 适度冗余说明}

为便于黑名单快速检查，`blacklist` 中保留 `id\_number` 和 `phone` 快照字段；为便于日志审计，`operation\_log` 中保留 `operator\_name`。这些字段属于面向审计和风险控制的可解释冗余，后续建表时应与业务规则保持一致。

\subsection{关系模式}

\paragraph{1. 关系模式说明}

关系模式采用如下记法：

`表名(主键, 普通字段, 外键字段 -> 被引用表.字段)`

主键统一为 `id`。外键字段在关系模式中使用 `->` 标明引用关系。所有表在物理实现时默认包含 `create\_time`、`update\_time`，需要保留历史和软删除能力的表包含 `deleted`。

\paragraph{2. 系统权限与组织关系模式}

`department(id, parent\_id -> department.id, dept\_code, dept\_name, leader\_user\_id -> sys\_user.id, phone, sort\_order, status, create\_time, update\_time, deleted)`\\
`sys\_user(id, username, password\_hash, real\_name, phone, email, department\_id -> department.id, user\_type, status, last\_login\_time, create\_time, update\_time, deleted)`\\
`sys\_role(id, role\_code, role\_name, role\_desc, status, create\_time, update\_time, deleted)`\\
`sys\_permission(id, permission\_code, permission\_name, permission\_type, parent\_id -> sys\_permission.id, route\_path, component\_path, api\_path, sort\_order, status, create\_time, update\_time, deleted)`\\
`sys\_user\_role(id, user\_id -> sys\_user.id, role\_id -> sys\_role.id, create\_time, update\_time, deleted)`\\
`sys\_role\_permission(id, role\_id -> sys\_role.id, permission\_id -> sys\_permission.id, create\_time, update\_time, deleted)`\\

\paragraph{3. 访客预约关系模式}

\noindent $\bullet$ 数据库：MySQL 8。\\
