# 05 概念结构设计

## 1. 设计目标

概念结构设计的任务是将需求分析阶段得到的业务对象、数据流、数据存储和处理过程抽象为数据库概念模型。本阶段不直接编写 SQL 建表语句，而是从“重庆邮电大学智慧访客预约与出入校管理系统”的业务语义出发，识别实体、属性、主键候选和实体之间的联系，为后续逻辑结构设计和 MySQL 表结构设计提供依据。

系统核心业务流程为：访客提交预约申请 → 系统检查黑名单 → 被访人确认 → 部门审批 → 生成通行凭证 → 门岗核验 → 入校登记 → 离校登记 → 访问记录归档 → 查询统计。概念模型必须能够覆盖该流程中的参与者、业务单据、审批轨迹、通行凭证、出入校记录、安全控制和统计支撑数据。

## 2. 实体识别方法

根据需求分析和数据字典，系统中的实体主要来源于以下几类对象：

1. 参与业务的人或组织：系统用户、角色、权限、部门、访客。
2. 校园出入管理的基础对象：校门、访客车辆、随行人员。
3. 核心业务单据：访客预约、审批记录、通行凭证、出入校记录、黑名单。
4. 系统运行支撑对象：通知消息、操作日志、字典类型、字典项。
5. 自动化交付支撑对象：截图记录、报告记录。

在概念结构中，用户与角色、角色与权限属于多对多联系，后续逻辑结构设计将转换为 `sys_user_role` 和 `sys_role_permission` 两个关系表。

## 3. 核心实体说明

| 实体 | 中文名称 | 主键 | 主要属性 | 存在原因 |
|---|---|---|---|---|
| sys_user | 系统用户 | id | 用户名、密码摘要、姓名、手机号、部门、用户类型、状态 | 表示被访人、审批人员、门岗安保、系统管理员和校级管理人员等校内系统使用者 |
| sys_role | 角色 | id | 角色编码、角色名称、角色说明、状态 | 支撑基于角色的权限控制和菜单控制 |
| sys_permission | 权限 | id | 权限编码、权限名称、权限类型、路由、接口路径、状态 | 描述系统菜单和接口权限，是 RBAC 授权的基本单位 |
| department | 部门 | id | 部门编码、部门名称、负责人、联系电话、状态 | 表示学校组织结构，支撑被访人归属、部门审批和统计分析 |
| campus_gate | 校门 | id | 校门编码、校门名称、位置、类型、状态 | 表示访客入校和离校发生的物理地点 |
| visitor | 访客 | id | 姓名、证件类型、证件号、手机号、单位、访客类型 | 表示校外来访人员，是预约和黑名单管理的基础对象 |
| visitor_vehicle | 访客车辆 | id | 访客、车牌号、车辆类型、颜色、品牌、状态 | 记录访客车辆信息，便于门岗核验和车辆通行管理 |
| visitor_companion | 随行人员 | id | 预约、姓名、证件类型、证件号、手机号、备注 | 表示随同访客进入校园的人员 |
| visit_apply | 访客预约 | id | 预约编号、访客、被访人、部门、访问时间、事由、预约状态、访问状态 | 表示访客入校申请，是业务流程的核心单据 |
| approval_record | 审批记录 | id | 预约、审批环节、审批人、审批结果、审批意见、审批时间 | 记录被访人确认和部门审批全过程，保证审批可追溯 |
| pass_code | 通行凭证 | id | 预约、通行码、有效期、凭证状态、使用时间、核验次数 | 表示审批通过后允许门岗核验的数字凭证 |
| access_record | 出入校记录 | id | 预约、访客、通行凭证、入校校门、离校校门、入校时间、离校时间、访问状态 | 记录访客实际进入和离开校园的过程 |
| blacklist | 黑名单 | id | 访客、证件号、手机号、原因、限制期限、状态、操作人 | 表示限制或提醒入校的风险访客信息 |
| notice | 通知消息 | id | 接收人、标题、内容、业务类型、业务编号、阅读状态 | 支撑预约、审批和系统事件的消息提醒 |
| operation_log | 操作日志 | id | 操作人、模块、操作类型、请求地址、结果、操作时间 | 记录关键系统操作，满足审计和问题追踪需要 |
| dict_type | 字典类型 | id | 字典编码、字典名称、说明、状态 | 统一管理状态、证件类型、车辆类型等基础分类 |
| dict_item | 字典项 | id | 字典类型、字典项编码、显示名称、值、排序、状态 | 表示某一字典类型下的具体可选值 |
| screenshot_record | 截图记录 | id | 截图编码、页面名称、路由、角色、文件路径、截图时间、状态 | 保存自动截图信息，用于报告素材管理 |
| report_record | 报告记录 | id | 报告编码、报告名称、Markdown 路径、Word 路径、生成状态、生成时间 | 保存自动报告生成结果和历史记录 |

## 4. 实体联系分析

| 联系名称 | 实体 A | 联系类型 | 实体 B | 业务含义 |
|---|---|---|---|---|
| 部门包含用户 | department | 1:N | sys_user | 一个部门包含多个校内用户，一个用户通常归属于一个部门 |
| 用户拥有角色 | sys_user | M:N | sys_role | 一个用户可以拥有多个角色，一个角色可以分配给多个用户 |
| 角色拥有权限 | sys_role | M:N | sys_permission | 一个角色可以拥有多个权限，一个权限可授予多个角色 |
| 访客提交预约 | visitor | 1:N | visit_apply | 一个访客可以提交多个预约申请 |
| 被访人接收预约 | sys_user | 1:N | visit_apply | 一个校内用户可作为被访人接收多个预约 |
| 部门关联预约 | department | 1:N | visit_apply | 预约根据被访人部门进入对应部门审批和统计 |
| 访客拥有车辆 | visitor | 1:N | visitor_vehicle | 一个访客可登记多辆车辆 |
| 预约使用车辆 | visitor_vehicle | 1:N | visit_apply | 一辆已登记车辆可被多个预约复用，每个预约最多选择一辆车辆 |
| 预约包含随行人员 | visit_apply | 1:N | visitor_companion | 一个预约申请可以包含多个随行人员 |
| 预约产生审批记录 | visit_apply | 1:N | approval_record | 一个预约会产生被访人确认和部门审批等多条记录 |
| 用户处理审批 | sys_user | 1:N | approval_record | 被访人或审批人员作为审批人产生审批记录 |
| 审批通过生成通行凭证 | visit_apply | 1:0..1 | pass_code | 只有审批通过的预约才生成一个有效通行凭证 |
| 预约产生出入校记录 | visit_apply | 1:N | access_record | 一个预约可以形成实际出入校记录，保留扩展多次通行能力 |
| 通行凭证用于出入校 | pass_code | 1:N | access_record | 门岗核验通过后使用通行凭证登记出入校 |
| 校门关联出入校记录 | campus_gate | 1:N | access_record | 一个校门可以关联多条入校或离校记录 |
| 访客关联黑名单 | visitor | 1:N | blacklist | 一个访客可能因不同原因被加入黑名单或保留历史记录 |
| 用户接收通知 | sys_user | 1:N | notice | 一个系统用户可以接收多条通知消息 |
| 用户产生操作日志 | sys_user | 1:N | operation_log | 一个用户在系统中可产生多条操作日志 |
| 字典类型包含字典项 | dict_type | 1:N | dict_item | 一个字典类型包含多个字典项 |
| 用户触发截图记录 | sys_user | 1:N | screenshot_record | 系统管理员或自动化任务记录截图结果 |
| 用户触发报告记录 | sys_user | 1:N | report_record | 系统管理员或自动化任务记录报告生成结果 |

## 5. 关键业务覆盖检查

| 业务环节 | 概念模型支撑 |
|---|---|
| 访客提交预约 | `visitor`、`visitor_vehicle`、`visitor_companion`、`visit_apply` |
| 黑名单检查 | `visitor`、`blacklist` |
| 被访人确认 | `sys_user`、`visit_apply`、`approval_record`、`notice` |
| 部门审批 | `department`、`sys_user`、`visit_apply`、`approval_record` |
| 通行凭证生成 | `visit_apply`、`pass_code` |
| 门岗核验 | `campus_gate`、`pass_code`、`visit_apply`、`blacklist`、`operation_log` |
| 入校登记 | `access_record`、`campus_gate`、`sys_user` |
| 离校登记 | `access_record`、`campus_gate`、`sys_user` |
| 当前在校和超时未离校 | `visit_apply`、`access_record` |

| 自动截图与报告 | `screenshot_record`、`report_record` |

## 6. 概念结构设计结论

本阶段识别的实体能够覆盖系统从预约提交、审批核验、出入校登记到统计归档的完整业务流程。用户、角色和权限实体能够支撑 RBAC 权限控制；预约、审批、通行凭证和出入校记录实体能够支撑核心业务状态流转；部门、校门、黑名单和日志实体能够支撑校园管理、安全控制和统计分析；截图记录和报告记录实体则支撑本课程设计要求中的自动截图与自动报告生成。

该概念模型不直接等同于 SQL 表结构，但已经为后续逻辑结构设计提供了清晰的实体、属性和联系基础。多对多联系将在逻辑设计阶段转换为关联表，实体属性将在 MySQL 表字段中进一步细化数据类型、约束、索引和默认值。
## 7. E-R 图表达策略

由于本系统同时包含访客预约主流程、RBAC 权限控制、消息日志、字典配置、截图与报告生成等多类实体，如果将全部实体和联系绘制在一张 E-R 图中，会导致图形过大、连线交叉严重，不利于课程设计报告阅读。因此，本项目在概念结构设计阶段采用“总体简化图 + 分域详图”的表达方式：

1. `er_overview.mmd` 用于展示系统主要实体和主干关系，适合放在报告正文开头说明整体概念模型。
2. `er_core_business.mmd` 用于展示访客预约、审批、通行码、门岗核验、出入校登记和黑名单检查等核心业务实体，适合作为正文重点图。
3. `er_user_permission.mmd` 用于展示部门、用户、角色、权限及关联表，说明系统的 RBAC 权限模型。
4. `er_system_support.mmd` 用于展示通知、操作日志、数据字典、截图记录和报告记录等支撑实体。
5. `table_relation.mmd` 用于展示后续逻辑结构中的主外键关系，适合作为其它设计图或附录。

这种拆分方式既保留了完整的概念设计信息，又避免在同一张图中强行展示所有实体造成的视觉混乱。后续逻辑结构设计仍然以完整实体清单为基础，多对多联系转换为关联表，弱关联和支撑性实体在表关系图中补充说明。
