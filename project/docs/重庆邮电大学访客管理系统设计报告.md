## 一、系统定义
## 1. 中文系统名称

重庆邮电大学智慧访客预约与出入校管理系统

## 2. 英文系统名称

Smart Visitor Reservation and Campus Access Management System of Chongqing University of Posts and Telecommunications

## 3. 系统设计意图

本系统面向高校校园访客管理场景，旨在通过数据库应用系统对校外访客从预约申请、校内确认、部门审批、门岗核验、入校登记、离校登记到记录归档和统计分析的全过程进行信息化管理。系统以 MySQL 数据库为核心，对访客信息、预约信息、审批信息、通行凭证、出入校记录、黑名单、用户角色权限和系统日志等数据进行统一组织、约束、查询和统计。

## 4. 系统目标

1. 建立规范化的访客预约和审批流程，减少线下沟通和纸质登记。
2. 通过黑名单检查、角色权限和门岗核验提升校园出入安全性。
3. 通过数据库记录保存完整审批链路和出入校轨迹，便于追溯和审计。
4. 通过统计报表分析访客数量、部门接待情况、校门通行情况和超时未离校情况。
5. 为数据库原理课程设计提供完整的需求分析、数据流程图、数据字典、E-R 图、关系模式、SQL 查询、系统实现和测试材料。

## 5. 系统服务对象

| 服务对象 | 说明 |
|---|---|
| 访客 | 校外来访人员，使用系统提交预约申请并查看预约状态和通行凭证 |
| 被访人 | 校内教师或工作人员，负责确认访客预约是否属实 |
| 部门审批人员 | 对本部门相关访问申请进行审核 |
| 门岗安保人员 | 在校门处核验访客通行凭证并办理入校、离校登记 |
| 系统管理员 | 维护用户、角色、权限、部门、校门、字典、黑名单和日志 |
| 校级管理人员 | 查看全校访客记录和统计报表，为管理决策提供依据 |

## 6. 系统核心功能

1. 访客预约申请：登记访客身份、联系方式、来访事由、被访人、访问时间、车辆和随行人员信息。
2. 黑名单检查：在提交预约和门岗核验环节检查访客是否存在限制入校记录。
3. 被访人确认：被访人对预约申请进行确认或拒绝。
4. 部门审批：部门审批人员对已确认预约进行审批。
5. 通行凭证生成：审批通过后生成通行码并记录有效期和使用状态。
6. 门岗核验：门岗安保人员核验通行码、访客身份和预约状态。
7. 入校登记与离校登记：记录访客进出校门、时间、经办人员和访问状态。
8. 当前在校与超时未离校管理：查询已入校未离校访客并识别超时风险。
9. 黑名单管理：维护黑名单原因、限制期限和处理状态。
10. 访客记录查询与统计报表：支持按访客、部门、校门、时间、状态等条件查询统计。
11. 系统管理：维护用户、角色权限、部门、校门、字典、通知和系统日志。
12. 自动截图与报告生成：为课程设计报告自动生成运行截图和文档素材。

## 7. 系统建设意义

从校园管理角度看，本系统能够提升访客预约审批效率，强化出入校安全控制，减少人工登记差错。从数据库课程设计角度看，本系统业务流程完整、数据对象丰富、实体关系清晰，适合展示数据库需求分析、概念结构设计、逻辑结构设计、SQL 查询设计和系统实现测试等核心能力。

## 二、需求分析
### 1. 综合需求
系统面向重庆邮电大学校外访客预约、校内被访人确认、部门审批、门岗核验、入校离校登记和校级统计监管等场景，要求实现从预约申请到访问记录归档的闭环管理。

### 2. 功能需求
## 1. 业务背景

高校日常教学、科研、合作交流和后勤服务中存在大量校外访客入校需求。访客通常需要提前说明来访事由、被访人、访问时间、车辆和随行人员等信息，学校则需要完成身份核验、被访人确认、部门审批、门岗放行、出入校记录留存和后续统计分析。随着校园安全管理要求提高，传统人工登记方式难以满足流程规范化、数据可追溯和统计自动化的需要。

## 2. 当前问题

1. 预约与审批过程分散，访客、被访人、审批人员和门岗之间信息传递效率低。
2. 纸质登记或临时登记难以保证数据完整性，后续查询统计成本高。
3. 黑名单、超时未离校、重复入校等风险场景缺少自动检查。
4. 门岗核验依赖人工判断，缺少与预约和审批结果联动的数据依据。
5. 访客数据缺少统一归档，不利于按部门、校门、时间和状态进行统计分析。

## 3. 系统目标

系统必须围绕“访客预约与出入校管理”建立统一的数据处理流程，实现预约申请、黑名单检查、被访人确认、部门审批、通行凭证生成、门岗核验、入校登记、离校登记、记录归档和查询统计。所有关键业务操作都应保存到数据库，并能在后续报告中通过表结构、关系模式和 SQL 查询体现。

## 4. 用户角色分析

| 角色 | 主要职责 | 数据访问范围 |
|---|---|---|
| 访客 | 提交预约申请、维护本人访客信息、查看预约状态和通行凭证 | 本人预约、本人通行凭证、本人出入校结果 |
| 被访人 | 查看与本人相关的预约申请，确认或拒绝访客来访 | 本人作为被访人的预约和审批记录 |
| 部门审批人员 | 审批本部门访客申请，填写审批意见 | 本部门预约、审批记录和统计数据 |
| 门岗安保人员 | 核验通行凭证，登记入校和离校，查看当前在校和超时未离校访客 | 门岗核验数据、出入校记录、通行凭证状态 |
| 系统管理员 | 管理用户、角色、权限、部门、校门、黑名单、字典、日志 | 系统全量管理数据 |
| 校级管理人员 | 查看全校访客记录和统计报表 | 全校访客记录和统计结果，通常只读 |

## 5. 核心业务流程

访客提交预约申请 → 系统检查黑名单 → 被访人确认 → 部门审批 → 生成通行凭证 → 门岗核验 → 入校登记 → 离校登记 → 访问记录归档 → 查询统计。

异常流程包括：黑名单访客被拦截、被访人拒绝、部门审批拒绝、通行凭证过期、重复入校登记、重复离校登记、已入校但超时未离校等。

## 7. 性能需求

1. 常规列表查询在测试数据规模下应在 2 秒内返回。
2. 门岗核验、入校登记和离校登记属于高频操作，应尽量减少人工输入。
3. 统计报表应支持按日期范围、部门和校门筛选，并能在课程演示数据规模下快速生成。
4. 关键查询字段如预约状态、访问时间、部门、校门、访客证件号应设计索引。

## 8. 安全需求

1. 系统必须提供登录认证，后续采用 JWT 进行接口访问控制。
2. 密码必须加密存储，不能以明文保存。
3. 不同角色只能访问授权菜单和接口。
4. 访客只能查看本人预约和通行凭证。
5. 部门审批人员只能审批本部门预约。
6. 门岗安保人员只能执行核验和出入校登记相关操作。
7. 关键操作必须写入系统日志，便于追溯。

## 9. 运行需求

1. 数据库运行环境为 MySQL 8。
2. 后端运行环境为 JDK 17 或兼容 Spring Boot 3 的环境。
3. 前端运行环境为 Node.js 与 Vite。
4. 系统应提供初始化数据库、启动后端、启动前端、自动截图和生成报告的脚本。
5. README 必须说明运行步骤、测试账号和常见问题。

## 10. 数据需求

系统需要维护以下数据：用户、角色、权限、部门、校门、访客、访客车辆、随行人员、预约申请、审批记录、通行凭证、出入校记录、黑名单、通知、操作日志、字典项、截图记录和报告记录。所有数据名称应在数据库表、后端实体、前端字段和报告中保持一致。

## 三、系统设计
### 1. 概念结构设计
## 1. 设计目标

概念结构设计的任务是将需求分析阶段得到的业务对象、数据流、数据存储和处理过程抽象为数据库概念模型。本阶段不直接编写 SQL 建表语句，而是从“重庆邮电大学智慧访客预约与出入校管理系统”的业务语义出发，识别实体、属性、主键候选和实体之间的联系，为后续逻辑结构设计和 MySQL 表结构设计提供依据。

系统核心业务流程为：访客提交预约申请 → 系统检查黑名单 → 被访人确认 → 部门审批 → 生成通行凭证 → 门岗核验 → 入校登记 → 离校登记 → 访问记录归档 → 查询统计。概念模型必须能够覆盖该流程中的参与者、业务单据、审批轨迹、通行凭证、出入校记录、安全控制和统计支撑数据。

## 2. 实体识别方法

根据需求分析和数据字典，系统中的实体主要来源于以下几类对象：

1. 参与业务的人或组织：系统用户、角色、权限、部门、访客。
2. 校园出入管理的基础对象：校门、访客车辆、随行人员。
3. 核心业务单据：访客预约、审批记录、通行凭证、出入校记录、黑名单。
4. 系统运行支撑对象：通知消息、操作日志、字典类型、字典项。
5. 自动化交付支撑对象：截图记录、报告记录。

在概念结构中，用户与角色、角色与权限属于多对多联系，后续逻辑结构设计将转换为 `sys_user_role` 和 `sys_role_permission` 两个关系表。

## 6. 概念结构设计结论

本阶段识别的实体能够覆盖系统从预约提交、审批核验、出入校登记到统计归档的完整业务流程。用户、角色和权限实体能够支撑 RBAC 权限控制；预约、审批、通行凭证和出入校记录实体能够支撑核心业务状态流转；部门、校门、黑名单和日志实体能够支撑校园管理、安全控制和统计分析；截图记录和报告记录实体则支撑本课程设计要求中的自动截图与自动报告生成。

该概念模型不直接等同于 SQL 表结构，但已经为后续逻辑结构设计提供了清晰的实体、属性和联系基础。多对多联系将在逻辑设计阶段转换为关联表，实体属性将在 MySQL 表字段中进一步细化数据类型、约束、索引和默认值。

### 3. 逻辑结构设计
## 1. 设计目标

逻辑结构设计是在概念结构设计基础上，将实体、属性和联系转换为适合 MySQL 8 实现的关系模式。本阶段重点明确表、字段、主键、外键、唯一约束、非空约束、状态字段、索引建议和范式检查，为后续 `schema.sql` 建表脚本提供直接依据。

## 2. 转换原则

1. 每个独立实体转换为一张关系表，表名统一使用小写下划线命名。
2. 主键统一使用 `id`，类型建议为 `BIGINT`。
3. 一对多联系通过在“多”的一方增加外键实现。
4. 多对多联系转换为关联表，例如 `sys_user_role`、`sys_role_permission`。
5. 时间字段统一使用 `create_time`、`update_time`，业务时间保留明确名称，如 `entry_time`、`approval_time`。
6. 如需软删除，统一使用 `deleted` 字段，`0` 表示未删除，`1` 表示已删除。
7. 状态字段统一使用 `VARCHAR(30)`，具体取值后续由字典表或业务常量维护。
8. 关键查询字段设置索引，保证预约查询、审批查询、门岗核验和统计报表效率。

## 4. 主要约束设计

| 约束类型 | 设计说明 |
|---|---|
| 主键约束 | 所有表统一使用 `id` 作为主键，便于后端实体继承和 MyBatis Plus 使用 |
| 外键约束 | 业务强关联字段设置外键，例如 `visit_apply.visitor_id`、`approval_record.apply_id`、`access_record.entry_gate_id` |
| 唯一约束 | 账号、角色编码、权限编码、部门编码、校门编码、预约编号、通行码等字段设置唯一约束 |
| 非空约束 | 业务必填字段设置非空，如姓名、手机号、状态、预约时间、审批结果等 |
| 默认值 | 状态字段设置合理默认值，如启用、待确认、未读、未删除 |
| 软删除 | 管理类和业务主表使用 `deleted` 字段，避免误删历史数据 |

## 5. 索引设计思路

1. 登录认证：`sys_user.username`、`sys_user.phone`。
2. 权限查询：`sys_user_role.user_id`、`sys_role_permission.role_id`。
3. 预约查询：`visit_apply.apply_no`、`visitor_id`、`host_user_id`、`department_id`、`apply_status`、`plan_start_time`。
4. 审批查询：`approval_record.apply_id`、`approver_user_id`、`approval_result`。
5. 门岗核验：`pass_code.pass_code`、`pass_status`、`valid_from`、`valid_to`。
6. 出入校管理：`access_record.apply_id`、`visitor_id`、`entry_gate_id`、`exit_gate_id`、`access_status`、`entry_time`、`exit_time`。
7. 黑名单检查：`blacklist.visitor_id`、`id_number`、`phone`、`status`。
8. 统计报表：围绕时间、部门、校门、状态等查询条件建立组合索引。

## 6. 范式检查

### 6.1 第一范式

各表字段均为原子字段，例如访客手机号、证件号、预约开始时间、审批结果、通行码等都不再拆分为重复组。随行人员和车辆信息没有直接以逗号拼接方式存储在预约表中，而是分别使用 `visitor_companion` 和 `visitor_vehicle` 表管理。

### 6.2 第二范式

所有业务表使用单字段主键 `id`，非主属性均依赖该主键。多对多关系通过关联表独立表示，避免用户角色和角色权限在主表中形成部分依赖。

### 6.3 第三范式

部门名称、校门名称、角色名称、权限名称等独立信息不在预约表、审批表和出入校记录表中重复保存，而是通过外键关联获取。这样可以减少冗余并保证基础数据修改后的一致性。

### 6.4 适度冗余说明

为便于黑名单快速检查，`blacklist` 中保留 `id_number` 和 `phone` 快照字段；为便于日志审计，`operation_log` 中保留 `operator_name`。这些字段属于面向审计和风险控制的可解释冗余，后续建表时应与业务规则保持一致。

### 4. 关系模式
## 1. 关系模式说明

关系模式采用如下记法：

`表名(主键, 普通字段, 外键字段 -> 被引用表.字段)`

主键统一为 `id`。外键字段在关系模式中使用 `->` 标明引用关系。所有表在物理实现时默认包含 `create_time`、`update_time`，需要保留历史和软删除能力的表包含 `deleted`。

## 2. 系统权限与组织关系模式

1. `department(id, parent_id -> department.id, dept_code, dept_name, leader_user_id -> sys_user.id, phone, sort_order, status, create_time, update_time, deleted)`
2. `sys_user(id, username, password_hash, real_name, phone, email, department_id -> department.id, user_type, status, last_login_time, create_time, update_time, deleted)`
3. `sys_role(id, role_code, role_name, role_desc, status, create_time, update_time, deleted)`
4. `sys_permission(id, permission_code, permission_name, permission_type, parent_id -> sys_permission.id, route_path, component_path, api_path, sort_order, status, create_time, update_time, deleted)`
5. `sys_user_role(id, user_id -> sys_user.id, role_id -> sys_role.id, create_time, update_time, deleted)`
6. `sys_role_permission(id, role_id -> sys_role.id, permission_id -> sys_permission.id, create_time, update_time, deleted)`

## 3. 访客预约关系模式

1. `visitor(id, visitor_name, id_type, id_number, phone, gender, unit_name, visitor_type, remark, create_time, update_time, deleted)`
2. `visitor_vehicle(id, visitor_id -> visitor.id, plate_number, vehicle_type, vehicle_color, brand, status, create_time, update_time, deleted)`
3. `visit_apply(id, apply_no, visitor_id -> visitor.id, host_user_id -> sys_user.id, department_id -> department.id, vehicle_id -> visitor_vehicle.id, visit_reason, plan_start_time, plan_end_time, apply_status, access_status, companion_count, submit_time, cancel_time, cancel_reason, remark, create_time, update_time, deleted)`
4. `visitor_companion(id, apply_id -> visit_apply.id, companion_name, id_type, id_number, phone, relation_remark, create_time, update_time, deleted)`

## 4. 审批、通行与出入校关系模式

1. `approval_record(id, apply_id -> visit_apply.id, approval_step, approver_user_id -> sys_user.id, approval_result, approval_comment, approval_time, sort_order, create_time, update_time, deleted)`
2. `pass_code(id, apply_id -> visit_apply.id, pass_code, qr_content, valid_from, valid_to, pass_status, used_time, verify_count, create_time, update_time, deleted)`
3. `access_record(id, apply_id -> visit_apply.id, visitor_id -> visitor.id, pass_code_id -> pass_code.id, entry_gate_id -> campus_gate.id, exit_gate_id -> campus_gate.id, entry_guard_id -> sys_user.id, exit_guard_id -> sys_user.id, entry_time, exit_time, access_status, overtime_flag, abnormal_reason, create_time, update_time, deleted)`
4. `campus_gate(id, gate_code, gate_name, gate_location, gate_type, status, remark, create_time, update_time, deleted)`

## 5. 风险控制、通知和日志关系模式

1. `blacklist(id, visitor_id -> visitor.id, id_number, phone, reason, level, start_time, end_time, status, operator_user_id -> sys_user.id, create_time, update_time, deleted)`
2. `notice(id, receiver_user_id -> sys_user.id, receiver_type, title, content, business_type, business_id, read_status, read_time, create_time, update_time, deleted)`
3. `operation_log(id, operator_user_id -> sys_user.id, operator_name, module_name, operation_type, request_method, request_url, operation_result, ip_address, operation_time, error_message, create_time, update_time, deleted)`

## 6. 字典与自动化材料关系模式

1. `dict_type(id, dict_code, dict_name, description, status, create_time, update_time, deleted)`
2. `dict_item(id, dict_type_id -> dict_type.id, item_code, item_label, item_value, sort_order, status, remark, create_time, update_time, deleted)`
3. `screenshot_record(id, screenshot_code, page_name, route_path, role_code, file_path, description, capture_time, status, created_by -> sys_user.id, create_time, update_time, deleted)`
4. `report_record(id, report_code, report_name, markdown_path, word_path, generate_status, generate_time, generated_by -> sys_user.id, error_message, create_time, update_time, deleted)`

## 7. 主要联系映射

| 概念联系 | 关系模式实现 |
|---|---|
| 一个部门包含多个系统用户 | `sys_user.department_id -> department.id` |
| 一个用户可以拥有多个角色 | `sys_user_role.user_id -> sys_user.id`, `sys_user_role.role_id -> sys_role.id` |
| 一个角色可以拥有多个权限 | `sys_role_permission.role_id -> sys_role.id`, `sys_role_permission.permission_id -> sys_permission.id` |
| 一个访客可以提交多个预约申请 | `visit_apply.visitor_id -> visitor.id` |
| 一个预约申请可以包含多个随行人员 | `visitor_companion.apply_id -> visit_apply.id` |
| 一个预约申请可以关联车辆信息 | `visit_apply.vehicle_id -> visitor_vehicle.id` |
| 一个预约申请可以产生多条审批记录 | `approval_record.apply_id -> visit_apply.id` |
| 审批通过预约可以生成一个通行凭证 | `pass_code.apply_id -> visit_apply.id`，后续设置唯一约束 |
| 一个预约申请可以产生出入校记录 | `access_record.apply_id -> visit_apply.id` |
| 一个校门可以关联多条出入校记录 | `access_record.entry_gate_id -> campus_gate.id`, `access_record.exit_gate_id -> campus_gate.id` |
| 一个访客可能被加入黑名单 | `blacklist.visitor_id -> visitor.id` |
| 一个用户可以产生多条操作日志 | `operation_log.operator_user_id -> sys_user.id` |

## 7. 按部门统计本月访客数量

查询目的：统计本月各部门预约量、独立访客数和审批通过数量。  
涉及数据表：`department`、`visit_apply`。  
SQL 语句：

```sql
SELECT
  d.dept_name,
  COUNT(va.id) AS monthly_apply_count,
  COUNT(DISTINCT va.visitor_id) AS unique_visitor_count,
  SUM(CASE WHEN va.apply_status = 'APPROVED' THEN 1 ELSE 0 END) AS approved_count
FROM department d
LEFT JOIN visit_apply va ON va.department_id = d.id
  AND va.submit_time >= DATE_FORMAT(CURRENT_DATE(), '%Y-%m-01')
  AND va.submit_time < DATE_ADD(DATE_FORMAT(CURRENT_DATE(), '%Y-%m-01'), INTERVAL 1 MONTH)
  AND va.deleted = 0
WHERE d.deleted = 0
GROUP BY d.id, d.dept_name
ORDER BY monthly_apply_count DESC, d.id ASC;
```

查询结果说明：返回部门维度排行数据，用于统计报表中的“部门访客排行”。

## 8. 按校门统计通行数量

查询目的：统计各校门入校、离校和总通行次数。  
涉及数据表：`campus_gate`、`access_record`。  
SQL 语句：

```sql
SELECT
  gate_name,
  SUM(entry_count) AS entry_count,
  SUM(exit_count) AS exit_count,
  SUM(entry_count + exit_count) AS total_passage_count
FROM (
  SELECT cg.gate_name, COUNT(ar.id) AS entry_count, 0 AS exit_count
  FROM campus_gate cg
  LEFT JOIN access_record ar ON ar.entry_gate_id = cg.id AND ar.deleted = 0
  GROUP BY cg.id, cg.gate_name
  UNION ALL
  SELECT cg.gate_name, 0 AS entry_count, COUNT(ar.id) AS exit_count
  FROM campus_gate cg
  LEFT JOIN access_record ar ON ar.exit_gate_id = cg.id AND ar.deleted = 0
  GROUP BY cg.id, cg.gate_name
) gate_stat
GROUP BY gate_name
ORDER BY total_passage_count DESC;
```

查询结果说明：返回校门通行排行，可用于门岗通行统计图表。

## 14. 查询近七天访客趋势

涉及数据表：`visit_apply`、`access_record`。  
SQL 语句：

```sql
WITH RECURSIVE recent_days AS (
  SELECT DATE_SUB(CURRENT_DATE(), INTERVAL 6 DAY) AS stat_date
  UNION ALL
  SELECT DATE_ADD(stat_date, INTERVAL 1 DAY)
  FROM recent_days
  WHERE stat_date < CURRENT_DATE()
)
SELECT
  rd.stat_date,
  COUNT(va.id) AS apply_count,
  SUM(CASE WHEN va.apply_status = 'APPROVED' THEN 1 ELSE 0 END) AS approved_count,
  COUNT(ar.id) AS entry_count
FROM recent_days rd
LEFT JOIN visit_apply va ON DATE(va.submit_time) = rd.stat_date AND va.deleted = 0
LEFT JOIN access_record ar ON ar.apply_id = va.id AND ar.deleted = 0
GROUP BY rd.stat_date
ORDER BY rd.stat_date ASC;
```

查询结果说明：返回连续日期趋势数据，用于 ECharts 折线图。

```sql
SELECT

  SUM(CASE WHEN approval_result = 'APPROVED' THEN 1 ELSE 0 END) AS approved_count,
```

### 典型查询脚本

以下 SQL 来自 `database/query_examples.sql`，覆盖历史预约、待确认、待审批、在校访客、超时未离校、部门排行、校门通行、黑名单、审批记录、车辆随行人员、操作日志、趋势和审批通过率等查询。

```sql
-- 重庆邮电大学智慧访客预约与出入校管理系统
-- Typical SQL query examples for database course report

USE cqupt_visitor_system;
SET NAMES utf8mb4;
SET collation_connection = 'utf8mb4_unicode_ci';

-- 1. 查询某访客的历史预约记录
SET @visitor_phone = '13900010001';
SELECT
  va.apply_no,
  v.visitor_name,
  v.phone,
  u.real_name AS host_name,
  d.dept_name,
  va.visit_reason,
  va.plan_start_time,
  va.plan_end_time,
  va.apply_status,
  va.access_status
FROM visit_apply va
JOIN visitor v ON va.visitor_id = v.id
JOIN sys_user u ON va.host_user_id = u.id
JOIN department d ON va.department_id = d.id
WHERE v.phone = @visitor_phone AND va.deleted = 0
ORDER BY va.submit_time DESC;

-- 2. 查询某被访人的待确认预约
SET @host_user_id = 2;
SELECT
  va.id,
  va.apply_no,
  v.visitor_name,
  v.company,
  v.phone,
  va.visit_reason,
  va.plan_start_time,
  va.plan_end_time,
  va.submit_time
FROM visit_apply va
JOIN visitor v ON va.visitor_id = v.id
WHERE va.host_user_id = @host_user_id
  AND va.apply_status = 'PENDING_HOST'
  AND va.deleted = 0
ORDER BY va.submit_time ASC;

-- 3. 查询某部门的待审批预约
SET @department_id = 2;
SELECT
  va.id,
  va.apply_no,
  d.dept_name,
  v.visitor_name,
  u.real_name AS host_name,
  va.visit_reason,
  va.plan_start_time,
  va.plan_end_time,
  va.submit_time
FROM visit_apply va
JOIN visitor v ON va.visitor_id = v.id
JOIN sys_user u ON va.host_user_id = u.id
JOIN department d ON va.department_id = d.id
WHERE va.department_id = @department_id
  AND va.apply_status IN ('HOST_CONFIRMED', 'PENDING_DEPT')
  AND va.deleted = 0
ORDER BY va.submit_time ASC;

-- 4. 查询当天已入校访客
SELECT
  ar.id AS access_record_id,
  va.apply_no,
  v.visitor_name,
  v.phone,
  d.dept_name,
  cg.gate_name AS entry_gate_name,
  guard.real_name AS entry_guard_name,
  ar.entry_time,
  ar.access_status
FROM access_record ar
JOIN visit_apply va ON ar.apply_id = va.id
JOIN visitor v ON ar.visitor_id = v.id
JOIN department d ON va.department_id = d.id
LEFT JOIN campus_gate cg ON ar.entry_gate_id = cg.id
LEFT JOIN sys_user guard ON ar.entry_guard_id = guard.id
WHERE ar.entry_time >= CURRENT_DATE()
  AND ar.entry_time < DATE_ADD(CURRENT_DATE(), INTERVAL 1 DAY)
  AND ar.deleted = 0
ORDER BY ar.entry_time DESC;

-- 5. 查询当天已入校但未离校访客
SELECT
  ar.id AS access_record_id,
  va.apply_no,
  v.visitor_name,
  v.phone,
  d.dept_name,
  cg.gate_name AS entry_gate_name,
  ar.entry_time,
  va.plan_end_time,
  TIMESTAMPDIFF(MINUTE, ar.entry_time, NOW()) AS stay_minutes
FROM access_record ar
JOIN visit_apply va ON ar.apply_id = va.id
JOIN visitor v ON ar.visitor_id = v.id
JOIN department d ON va.department_id = d.id
LEFT JOIN campus_gate cg ON ar.entry_gate_id = cg.id
WHERE ar.entry_time >= CURRENT_DATE()
  AND ar.entry_time < DATE_ADD(CURRENT_DATE(), INTERVAL 1 DAY)
  AND ar.exit_time IS NULL
  AND ar.deleted = 0
ORDER BY ar.entry_time DESC;

-- 6. 查询超时未离校访客
SELECT
  ar.id AS access_record_id,
  va.apply_no,
  v.visitor_name,
  v.phone,
  d.dept_name,
  va.plan_end_time,
  ar.entry_time,
  TIMESTAMPDIFF(MINUTE, va.plan_end_time, NOW()) AS overtime_minutes,
  ar.abnormal_reason
FROM access_record ar
JOIN visit_apply va ON ar.apply_id = va.id
JOIN visitor v ON ar.visitor_id = v.id
JOIN department d ON va.department_id = d.id
WHERE ar.exit_time IS NULL
  AND va.plan_end_time < NOW()
  AND ar.deleted = 0
ORDER BY overtime_minutes DESC;

-- 7. 按部门统计本月访客数量
SELECT
  d.dept_name,
  COUNT(va.id) AS monthly_apply_count,
  COUNT(DISTINCT va.visitor_id) AS unique_visitor_count,
  SUM(CASE WHEN va.apply_status = 'APPROVED' THEN 1 ELSE 0 END) AS approved_count
FROM department d
LEFT JOIN visit_apply va ON va.department_id = d.id
  AND va.submit_time >= DATE_FORMAT(CURRENT_DATE(), '%Y-%m-01')
  AND va.submit_time < DATE_ADD(DATE_FORMAT(CURRENT_DATE(), '%Y-%m-01'), INTERVAL 1 MONTH)
  AND va.deleted = 0
WHERE d.deleted = 0
GROUP BY d.id, d.dept_name
ORDER BY monthly_apply_count DESC, d.id ASC;

-- 8. 按校门统计通行数量
SELECT
  gate_name,
  SUM(entry_count) AS entry_count,
  SUM(exit_count) AS exit_count,
  SUM(entry_count + exit_count) AS total_passage_count
FROM (
  SELECT cg.gate_name, COUNT(ar.id) AS entry_count, 0 AS exit_count
  FROM campus_gate cg
  LEFT JOIN access_record ar ON ar.entry_gate_id = cg.id AND ar.deleted = 0
  GROUP BY cg.id, cg.gate_name
  UNION ALL
  SELECT cg.gate_name, 0 AS entry_count, COUNT(ar.id) AS exit_count
  FROM campus_gate cg
  LEFT JOIN access_record ar ON ar.exit_gate_id = cg.id AND ar.deleted = 0
  GROUP BY cg.id, cg.gate_name
) gate_stat
GROUP BY gate_name
ORDER BY total_passage_count DESC;

-- 9. 查询黑名单访客
SELECT
  b.id AS blacklist_id,
  COALESCE(v.visitor_name, '未建档访客') AS visitor_name,
  b.id_number,
  b.phone,
  b.reason,
  b.level,
  b.start_time,
  b.end_time,
  b.status,
  u.real_name AS operator_name
FROM blacklist b
LEFT JOIN visitor v ON b.visitor_id = v.id
LEFT JOIN sys_user u ON b.operator_user_id = u.id
WHERE b.status = 'ACTIVE'
  AND b.deleted = 0
ORDER BY b.level DESC, b.start_time DESC;

-- 10. 查询某预约的完整审批记录
SET @apply_no = CONCAT('VA', DATE_FORMAT(CURRENT_DATE(), '%Y%m%d'), '0001');
SELECT
  va.apply_no,
  ar.approval_step,
  ar.approval_result,
  ar.approval_comment,
  ar.approval_time,
  u.real_name AS approver_name,
  d.dept_name AS approver_department
FROM approval_record ar
JOIN visit_apply va ON ar.apply_id = va.id
JOIN sys_user u ON ar.approver_user_id = u.id
LEFT JOIN department d ON u.department_id = d.id
WHERE va.apply_no = @apply_no
  AND ar.deleted = 0
ORDER BY ar.sort_order ASC, ar.approval_time ASC;

-- 11. 查询某访客的车辆和随行人员
SET @apply_id = 1;
SELECT
  va.apply_no,
  v.visitor_name,
  vv.plate_no,
  vv.vehicle_type,
  vv.brand,
  vc.companion_name,
  vc.phone AS companion_phone,
  vc.relation_remark
FROM visit_apply va
JOIN visitor v ON va.visitor_id = v.id
LEFT JOIN visitor_vehicle vv ON va.vehicle_id = vv.id
LEFT JOIN visitor_companion vc ON vc.apply_id = va.id AND vc.deleted = 0
WHERE va.id = @apply_id
  AND va.deleted = 0
ORDER BY vc.id ASC;

-- 12. 查询系统操作日志
SELECT
  ol.id,
  ol.operator_name,
  ol.module_name,
  ol.operation_type,
  ol.request_method,
  ol.request_url,
  ol.operation_result,
  ol.ip_address,
  ol.operation_time,
  ol.error_message
FROM operation_log ol
WHERE ol.operation_time >= DATE_ADD(NOW(), INTERVAL -7 DAY)
  AND ol.deleted = 0
ORDER BY ol.operation_time DESC
LIMIT 50;

-- 13. 查询今日访客概览
SELECT
  COUNT(DISTINCT va.id) AS today_apply_count,
  SUM(CASE WHEN va.apply_status = 'APPROVED' THEN 1 ELSE 0 END) AS today_approved_count,
  COUNT(DISTINCT ar.id) AS today_entry_count,
  SUM(CASE WHEN ar.exit_time IS NULL AND ar.entry_time IS NOT NULL THEN 1 ELSE 0 END) AS current_in_campus_count,
  SUM(CASE WHEN ar.exit_time IS NULL AND va.plan_end_time < NOW() THEN 1 ELSE 0 END) AS overtime_count
FROM visit_apply va
LEFT JOIN access_record ar ON ar.apply_id = va.id AND ar.deleted = 0
WHERE va.submit_time >= CURRENT_DATE()
  AND va.submit_time < DATE_ADD(CURRENT_DATE(), INTERVAL 1 DAY)
  AND va.deleted = 0;

-- 14. 查询近七天访客趋势
WITH RECURSIVE recent_days AS (
  SELECT DATE_SUB(CURRENT_DATE(), INTERVAL 6 DAY) AS stat_date
  UNION ALL
  SELECT DATE_ADD(stat_date, INTERVAL 1 DAY)
  FROM recent_days
  WHERE stat_date < CURRENT_DATE()
)
SELECT
  rd.stat_date,
  COUNT(va.id) AS apply_count,
  SUM(CASE WHEN va.apply_status = 'APPROVED' THEN 1 ELSE 0 END) AS approved_count,
  COUNT(ar.id) AS entry_count
FROM recent_days rd
LEFT JOIN visit_apply va ON DATE(va.submit_time) = rd.stat_date AND va.deleted = 0
LEFT JOIN access_record ar ON ar.apply_id = va.id AND ar.deleted = 0
GROUP BY rd.stat_date
ORDER BY rd.stat_date ASC;

-- 15. 查询审批通过率
SELECT
  COUNT(*) AS total_finished_approval_count,
  SUM(CASE WHEN approval_result = 'APPROVED' THEN 1 ELSE 0 END) AS approved_count,
  SUM(CASE WHEN approval_result = 'REJECTED' THEN 1 ELSE 0 END) AS rejected_count,
  ROUND(SUM(CASE WHEN approval_result = 'APPROVED' THEN 1 ELSE 0 END) / COUNT(*) * 100, 2) AS approval_rate_percent
FROM approval_record
WHERE approval_result IN ('APPROVED', 'REJECTED')
  AND deleted = 0;
```

|---|---|---|---|---|

```json
{
  "username": "admin",
  "password": "123456"
}
```

登录成功响应中 `data.token` 即 JWT Token，前端后续请求应加入请求头：

```http
Authorization: Bearer <token>
```

```json
{
  "comment": "来访事由属实，同意进入下一环节"
}
```

被访人确认后状态变为 `HOST_CONFIRMED`；部门审批通过后状态变为 `APPROVED`，系统自动生成 `pass_code`。

### 9.3 门岗核验示例

```json
{
  "passCode": "PC202606220001",
  "gateId": 1
}
```

也可以使用以下任一条件核验：`applyId`、`applyNo`、`phone`、`idNumber`、`passCode`。核验返回字段包括 `allowed`、`message`、`applyStatus`、`accessStatus`、`validFrom`、`validTo`。未审批、已拒绝、已取消、黑名单、过期凭证、重复入校都会返回 `allowed = false` 和明确原因。

### 9.4 入校和离校示例

入校登记：

```json
{
  "applyId": 8,
  "gateId": 1
}
```

离校登记：

```json
{
  "applyId": 8,
  "gateId": 2
}
```

入校成功后写入 `access_record.entry_time`，预约访问状态更新为 `ENTERED`。离校成功后写入 `access_record.exit_time`，预约访问状态更新为 `EXITED`。若重复离校，接口返回错误：`该访客已登记离校，不能重复离校`。

## 六、总结
本系统围绕重庆邮电大学访客管理场景，完成了从需求分析、概念结构设计、逻辑结构设计、MySQL 建库建表、后端接口、角色权限、访客预约审批流程、门岗核验、统计报表、自动截图到课程设计报告自动生成的完整闭环。数据库设计覆盖用户、角色、权限、部门、校门、访客、车辆、随行人员、预约申请、审批记录、通行凭证、出入校记录、黑名单、通知、日志、截图和报告等核心数据对象，能够支撑课程设计要求中的数据流程图、数据字典、E-R 图、关系模式、SQL 查询和系统测试内容。
